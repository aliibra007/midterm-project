# Midterm Project — Full-Stack School Portal

A complete school management portal built with **React** on the frontend and **ASP.NET Core 10** on the backend, using **PostgreSQL** as the database. Teachers can create classes and manage their student rosters. Students can browse and enroll in classes. Everything is protected by JWT authentication with an automatic logout after one hour of inactivity.

## How to Run It

### 1. Make sure PostgreSQL is running

The connection string is in `backend.api/appsettings.json`:

```json
"DefaultConnection": "Host=localhost;Port=5432;Database=schooldb;Username=postgres;Password=yourpassword"
```

Change the password to match your PostgreSQL setup if needed.

### 2. Start the backend

```bash
cd "backend.api"
dotnet run
```

The API will be available at `http://localhost:5200`. You can browse the auto-generated docs at `http://localhost:5200/swagger`.

### 3. Start the frontend

```bash
cd "ali_ibrahim"
npm install   # only needed the first time
npm run dev
```

Open `http://localhost:5173` in your browser.

### 4. Create your first accounts

The database starts empty. Use the register endpoints (via Postman or the Swagger UI) to create accounts:

```
POST http://localhost:5200/api/auth/register/teacher
{
  "name": "Ali Ibrahim",
  "email": "teacher@example.com",
  "password": "password123"
}

POST http://localhost:5200/api/auth/register/student
{
  "name": "Sara Nader",
  "email": "student@example.com",
  "password": "password123",
  "major": "Computer Science"
}
```

Then log in from the browser at `http://localhost:5173`.

## The Database Schema

Four tables, straightforward relationships:

```
Teacher ──< Class ──< Enrollment >── Student
```

- A **Teacher** owns many **Classes**
- A **Class** can have many **Students** (via **Enrollment**)
- A **Student** can be in many **Classes** (via **Enrollment**)
- The `Enrollment` table has a unique index on `(ClassId, StudentId)` so a student can't accidentally enroll twice

### Entity: `Student`
| Column       | Type         | Notes                        |
|-------------|--------------|------------------------------|
| Id           | int (PK)     | auto-increment               |
| Name         | varchar(100) | required                     |
| Email        | varchar(200) | required, unique             |
| PasswordHash | text         | BCrypt hash — never exposed  |
| Major        | varchar(100) | default: "Computer Science"  |

### Entity: `Teacher`
| Column       | Type         | Notes            |
|-------------|--------------|------------------|
| Id           | int (PK)     |                  |
| Name         | varchar(100) | required         |
| Email        | varchar(200) | required, unique |
| PasswordHash | text         | BCrypt hash      |

### Entity: `Class`
| Column      | Type         | Notes                         |
|------------|--------------|-------------------------------|
| Id          | int (PK)     |                               |
| Name        | varchar(200) | required                      |
| Schedule    | timestamp    | when the class meets          |
| MaxStudents | int          | 0 = no cap                    |
| TeacherId   | int (FK)     | references Teacher            |

### Entity: `Enrollment`
| Column     | Type      | Notes                    |
|-----------|-----------|--------------------------|
| Id         | int (PK)  |                          |
| ClassId    | int (FK)  | references Class         |
| StudentId  | int (FK)  | references Student       |
| EnrolledAt | timestamp | set automatically at insert |

All endpoints except `/api/auth/*` require a JWT in the `Authorization: Bearer <token>` header.

## Security

### JWT Authentication

When you log in successfully, the backend generates a JWT token that looks like this (decoded):

```json
{
  "nameid": "1",
  "email": "teacher@example.com",
  "role": "Teacher",
  "exp": 1700000000
}
```

The frontend stores this token in `localStorage` and attaches it to every API request:

```
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

Every controller is decorated with `[Authorize]`, so if the token is missing or invalid the backend returns `401 Unauthorized` before your code even runs.

### Role-Based Access

Some endpoints go further with `[Authorize(Roles = "Teacher")]` or `[Authorize(Roles = "Student")]`. If you're the wrong role, you get `403 Forbidden`.

On top of that, teachers can only edit or delete **their own** classes. If a teacher tries to touch another teacher's class, the controller checks the owner and returns `403 Forbidden`.

### Inactivity Logout (1 Hour)

The frontend sets up an event listener for mouse moves, key presses, clicks, scrolls, and touch events. Every time one of these fires, a one-hour countdown resets. If an hour passes with none of those events, the app:

1. Clears `localStorage` (removes the token)
2. Redirects to the login page

```js
// inside App.jsx
const INACTIVITY_LIMIT = 60 * 60 * 1000; // one hour in milliseconds

const resetTimer = useCallback(() => {
  clearTimeout(timerRef.current);
  timerRef.current = setTimeout(logout, INACTIVITY_LIMIT);
}, [logout]);

useEffect(() => {
  if (!user) return;
  const events = ['mousemove', 'mousedown', 'keydown', 'scroll', 'touchstart'];
  events.forEach(e => window.addEventListener(e, resetTimer));
  resetTimer();
  return () => {
    events.forEach(e => window.removeEventListener(e, resetTimer));
    clearTimeout(timerRef.current);
  };
}, [user, resetTimer]);
```

### Input Validation & SQL Injection Prevention

- All DTOs use `[Required]`, `[MaxLength]`, `[EmailAddress]`, `[MinLength]`, and `[Range]` attributes. If the input doesn't pass, the controller returns `400 Bad Request` before touching the database.
- All database access goes through Entity Framework Core, which uses parameterized queries under the hood — raw SQL strings concatenated with user input are never used, so SQL injection isn't possible.
- Passwords are hashed with **BCrypt** before being stored. The plain-text password never touches the database.

### Session Persistence Across Refreshes

The user object (token, role, name, email) is saved in `localStorage` on login. When the app loads, it reads from there first — so refreshing the page doesn't force a re-login. On logout (manual or inactivity), everything is cleared.

## Frontend Architecture

### `api.js` — Single Source of Truth for HTTP Calls

Rather than writing `fetch(...)` with headers everywhere, every network request goes through `api.js`:

```js
// token is read from localStorage each time a request is made
const authHeaders = () => ({
  "Content-Type": "application/json",
  Authorization: `Bearer ${localStorage.getItem("token")}`,
});

export const getMyClasses = () =>
  fetch(`${BASE}/class/my`, { headers: authHeaders() });
```

If you ever need to change the base URL or add a global header, you only change one file.

### Role Normalisation

The backend returns `"Role": "Teacher"` or `"Role": "Student"` (capitalised). On login, the frontend normalises this to lowercase before saving:

```js
const saveUser = (userData) => {
  const normalized = { ...userData, role: userData.role.toLowerCase() };
  localStorage.setItem('user', JSON.stringify(normalized));
  localStorage.setItem('token', userData.token);
  setuser(normalized);
};
```

This way the Sidebar's role checks (`user.role === "teacher"`) stay consistent.

## Common Issues

**"Could not reach the server"** — The backend isn't running. Start it with `dotnet run` in the `backend.api` folder.

**401 on all requests after login** — Your token expired (24h) or something went wrong saving it. Log out and back in.

**403 on teacher endpoints** — You're logged in as a student trying to hit a teacher-only endpoint, or you're trying to modify another teacher's class.

**Database connection failed** — Check the `DefaultConnection` string in `appsettings.json`. Make sure PostgreSQL is running and the credentials match.

**`dotnet ef` not found** — Install the tool: `dotnet tool install --global dotnet-ef`
